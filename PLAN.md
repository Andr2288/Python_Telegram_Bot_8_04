# План роботи над проєктом (handoff для агентів / розробників)

Документ описує **джерело вимог**, **що вже зроблено**, **наступні кроки** і **пастки середовища**, щоб наступні сесії могли продовжити без втрати контексту.

---

## 1. Авторитетні вимоги

- Повний текст ТЗ: **`Вимоги до боту.txt`** (корінь репозиторію).
- Додаткові вказівки до стилю роботи з кодом (якщо релевантно): `backend_reqs.txt`.

**Важливо по стеку (з ТЗ п.5):** мова — **Python**; бібліотека бота — **aiogram** або **python-telegram-bot**. Проєкт використовує **python-telegram-bot** (не Telegraf — це Node.js і суперечить ТЗ).

## 2. Поточна структура проєкту

```
Python_Telegram_Bot_8_04/
├── main.py              # точка входу, handlers / error handler / run_polling
├── config.py            # BOT_TOKEN з .env (обов’язково)
├── requirements.txt
├── .env                 # локально, не комітити (є .env.example)
├── .env.example
├── handlers/
│   ├── add.py           # ConversationHandler /add (структуровано)
│   └── __init__.py
├── helpers/
│   ├── parsing.py       # дата/час для /add
│   ├── user_context.py  # ensure_telegram_user
│   └── __init__.py
├── database/
│   ├── __init__.py
│   ├── activity.py      # log_activity → activity_log
│   ├── reminders.py     # insert_reminder
│   ├── users.py         # get_or_create_user, timezone
│   └── db.py            # SQLite: users, reminders, activity_log
├── reminders.db         # створюється після першого init_db()
├── Вимоги до боту.txt
└── PLAN.md              # цей файл
```

Схема БД уже закладена в `database/db.py` (таблиці `users`, `reminders`, `activity_log`). Поле `repeat_rule` у `reminders` — для майбутніх повторів.

## 5. Технічні нотатки (щоб не повторювати помилки)

### 5.1. Один екземпляр polling

Помилка **`409 Conflict`** означає: той самий `BOT_TOKEN` вже отримує `getUpdates` в іншому процесі або через webhook. Рішення: зупинити всі дублікати `main.py`, перевірити фонові Run у IDE.

### 5.2. Python 3.14+

Не прибирати **`_prepare_asyncio()`** без перевірки: без нього на нових Python можливий `RuntimeError: There is no current event loop`.

### 5.3. Час і часові пояси

- Зберігати в БД однозначно (рекомендовано **UTC ISO 8601** у текстовому полі або unix timestamp + правила).
- Показувати користувачу в його **`users.timezone`** (IANA).

### 5.4. Безпека

- Не логувати повні URL Telegram API на рівні INFO (токен у шляху). За потреби залишати `httpx` на WARNING або додати фільтр логів.

### 5.5. Залежності

Перед новими фічами оновлювати **`requirements.txt`** (наприклад `dateparser`, якщо оберете його для природної мови).

## 6. Критерії приймання (чеклист з ТЗ п.7)

Використовувати як фінальну перевірку перед здачею:

1. Бот стартує без помилок.
2. Два формати створення нагадування (структурований + текстовий).
3. Коректне визначення дати/часу з фраз.
4. Уточнення при неповному вводі.
5. Сповіщення в Telegram у потрібний час.
6. Повторювані нагадування (день / тиждень / місяць).
7. Часові пояси на користувача.
8. Перегляд, редагування, скасування, пошук, фільтрація.
9. SQLite з переліченими сутностями (користувачі, нагадування, статуси, повтор, лог).
10. Базова статистика.
11. Логування дій і зрозумілі помилки для користувача.

- Нові handlers краще виносити в окремі модулі (наприклад `bot/handlers/…`) і підключати в `main.py`, коли `main.py` стане великим — це відповідає «структурованому коду» з п.8 ТЗ (бажано).
